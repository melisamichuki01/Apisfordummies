01 What is an API?

Before we touch a single line of code, let's build a real picture of what an API actually is. The letters stand for Application Programming Interface — and yes, that sounds like a lot. But the idea is simple enough that we'll get it in the next two minutes.

🔁 Think of it like a restaurant

Imagine you walk into a restaurant. You don't go into the kitchen and cook your own food. You sit down, look at a menu, and give your order to the waiter. The waiter goes to the kitchen, collects your food, and brings it back to you.

The waiter is the API. You are the application (or your Python script). The kitchen is the server holding the data. The menu is the list of things you're allowed to ask for.

You never need to know how the kitchen works — you just need to know how to talk to the waiter. That's the whole point of an API.

In technical terms: an API is a set of rules that lets one program talk to another. When your Python script sends a request to an API, the API's server receives it, processes it, and sends back a response — usually as data in a format called JSON.

💡 What is JSON, really?

JSON stands for JavaScript Object Notation. Don't let the name scare you — it's just a way of packaging up data so it can travel across the internet cleanly.

It looks exactly like a Python dictionary. If you've written {"name": "Mel", "city": "Nairobi"} in Python — you already understand JSON. Curly braces, keys, and values. That's it.

02 Why data scientists need to know this

You might be wondering: "I work with data, not software. Do I really need to understand APIs?" The honest answer is yes — and here's exactly why.

Most real-world data doesn't sit in a nice CSV file waiting for you. It lives inside systems — databases, platforms, services — and the only way to get it out is through an API. Here are situations you will actually run into:

  • Pulling weather data for a forecasting model
  • Fetching financial market prices for analysis
  • Pulling social media data for sentiment analysis
  • Sending data to an ML model hosted on a cloud service
  • Connecting your dashboard to a live data source
  • Automating data ingestion as part of a pipeline
Bottom line: APIs are how the world shares data. If you want to work with real, live data — you need to know how to request it, receive it, and handle it.

03 HTTP — the language of the web

Before we can talk about REST APIs, we need to understand HTTP. This is the part most tutorials skip, and it's the reason a lot of beginners feel confused later. Let's take five minutes and do it properly.

What is HTTP?

HTTP stands for HyperText Transfer Protocol. A "protocol" is just a set of rules for how two things communicate. HTTP is specifically the rulebook that governs how computers talk to each other over the internet. Every time you visit a website, an HTTP request is happening behind the scenes.

🔁 In plain life terms...

Think of HTTP like the rules of a phone call. When you call someone: one person speaks, the other listens, then they reply. You don't both talk at once. There's a structure to the conversation.

HTTP works the same way. Your computer (the client) makes a request. The other computer (the server) sends back a response. The whole exchange follows the HTTP rulebook — one message at a time.

Why do we need to understand HTTP theory?

Here's the honest reason: when something goes wrong with an API call — and it will — you need to be able to read the error. HTTP errors have specific codes and meanings. If you don't know that a 401 means "you didn't authenticate properly" or a 429 means "you sent too many requests too fast," you'll waste hours debugging blind.

Understanding HTTP theory also helps you write better requests, handle edge cases properly, and build data pipelines that don't fail silently at 3am.

The request-response cycle

Every single API call follows this same cycle, every single time:

Your Python Script (the client)
⬇
sends an HTTP request
⬇
The Internet routes it to the right server
⬇
The API Server receives it, processes it
⬇
sends back an HTTP response (with data)
⬇
Your Python Script receives JSON and uses the data

What is inside an HTTP request?

Every request has four parts. You don't need to memorise all of them right now — just know they exist so nothing surprises you later.

PartWhat it isReal example
URL The address of what you want https://api.weather.com/v1/current
Method What you want to do GET — fetch data
Headers Extra info about the request API key, content type, auth token
Body Data you're sending (POST/PUT only) JSON data, form fields

What is inside an HTTP response?

PartWhat it tells you
Status code Did it work? 200 = yes, 404 = not found, 500 = server error
Headers Metadata about the response (content type, rate limit info, etc.)
Body The actual data — usually JSON
🚨 Hold on — what are "headers" exactly?

Headers are extra information attached to a request or response — like a sticky note on a letter. The sticky note might say: "This is urgent" or "I'm from company X" or "The content is in JSON format."

In API calls, the most important header you'll use is the Authorization header — that's where your API key goes to prove you're allowed to access the service. Another common one is Content-Type: application/json, which tells the server that the data you're sending is in JSON format.

You don't set most headers manually — the requests library handles them. But you'll often need to add your API key header by hand.

04 REST — a set of rules for building APIs

REST stands for Representational State Transfer. You absolutely do not need to memorise that. Just know that REST is a style of building APIs — a set of rules that, when followed, make an API predictable and easy to use. When people say "REST API" or "RESTful API," they mean an API built with these rules in mind.

🔁 Real world version

Imagine every post office in the world agreed to follow the same rules: packages always go in the same format, the label is always in the same position, tracking numbers always look the same. Once you've shipped a package once, you know how to ship from any post office anywhere.

REST is that agreement for APIs. Once you know how REST works, you can pick up almost any REST API in the world — even ones you've never seen before — because they all follow the same patterns.

REST was defined by Roy Fielding in his PhD dissertation in the year 2000. He laid out a set of constraints — a kind of constitution for how APIs should behave. APIs that follow these constraints are called "RESTful."

05 REST architecture — the six constraints

REST has six architectural constraints. These aren't abstract theory — each one has a direct impact on how you write API calls and how your pipelines behave. Let's go through them one at a time.

1. Client-Server

The client (your script) and the server (the API) are completely separate things. They don't need to know anything about each other's internal workings. Your script just sends requests; the server just handles them and responds. This separation means you can change your script without breaking the server, and vice versa.

2. Stateless

This is one of the most important constraints. "Stateless" means the server does not remember you between requests. Every single request you send must include all the information the server needs to respond — your API key, your query, everything. The server treats each request as if it has never seen you before.

🚨 Don't worry — this tripped everyone up the first time

Stateless sounds annoying, but it's actually a feature. Because the server doesn't have to remember anything about you, it can handle millions of requests from different users simultaneously without getting confused. Your requests can also be processed by any server in a cluster — not just one specific machine.

The practical impact on your code: always include your authentication header and all necessary parameters in every single request. Never assume the API "remembers" anything from your previous call.

3. Cacheable

Responses can be marked as cacheable, meaning the client can store a response and reuse it instead of making the same request again. This speeds things up and reduces load on the API server. In data pipelines, caching matters when you're calling the same endpoint repeatedly in a loop — it can make your work significantly faster and cheaper if the API charges per call.

4. Uniform Interface

This is the defining feature of REST. Every REST API uses the same interface structure: standard HTTP methods (GET, POST, PUT, DELETE), standard URL patterns, and standard response formats. This is why once you know one REST API, you can pick up another very quickly.

💡 Let me explain what "uniform interface" means in practice

Let's say you're working with two completely different APIs — one for weather data, one for financial data. Both follow REST. This means both will use:

  • A URL with a resource path (e.g., /v1/stocks or /v1/weather)
  • GET to fetch data, POST to send data
  • JSON in the response body
  • Standard status codes to tell you if it worked

You don't have to learn an entirely new communication system for each API. The interface is uniform — predictable across all of them.

5. Layered System

When you send a request to an API, you might think you're talking directly to one server. In reality, your request might pass through load balancers, caches, security gateways, and multiple servers before a response comes back. REST allows this layering — and you don't need to know or care about any of those layers. Your code stays exactly the same regardless.

6. Code on Demand (optional)

Servers can optionally send executable code (like JavaScript) back to the client. This constraint is rarely relevant in data science work, so we won't dwell on it. Just know it exists as part of the spec.

The two you'll think about most: Stateless and Uniform Interface. These two constraints shape how you write every single API call in a data pipeline.

06 Anatomy of an API request

Let's break down the URL of a real REST API call so you can read one with confidence and build your own without guessing.

Example API URL — fully annotated
https://api.openweathermap.org/data/2.5/weather?q=Nairobi&appid=YOUR_KEY&units=metric

Breaking it down piece by piece:

https://                     the protocol — always HTTPS for secure APIs
api.openweathermap.org       the host / domain name of the API server
/data/2.5/weather            the endpoint path — the specific resource we want
?                            marks the start of the query parameters
q=Nairobi                    query param: city we want weather for
&appid=YOUR_KEY              query param: our API key for authentication
&units=metric                query param: return temperature in Celsius

What is an endpoint?

An endpoint is a specific URL path that represents one particular resource or action in the API. Think of it as a specific page on a website — each page has a different address and returns different content.

Endpoint pathWhat it returns
/data/2.5/weatherCurrent weather for a city
/data/2.5/forecast5-day weather forecast
/data/2.5/air_pollutionAir quality index for coordinates

Base URL vs endpoint path

The base URL is the root address of the API: https://api.openweathermap.org. The endpoint path is what you add to it: /data/2.5/weather. Together they form the full request URL. In your code, you'll almost always store the base URL as a constant and build the full URL dynamically.

07 HTTP methods — the verbs of the web

HTTP methods tell the server what action you want to perform. In everyday data science work, you'll mostly use GET and POST. But knowing all of them helps you read API documentation confidently.

MethodWhat it doesEveryday analogyHas a body?
GET Fetch / read data Reading a menu No
POST Send data / create something Placing an order Yes
PUT Replace something completely Replacing your entire order Yes
PATCH Update part of something Changing just the drink Yes
DELETE Remove something Cancelling the order Sometimes
💡 A bit more on GET vs POST for data scientists

You'll use GET the most — you're usually pulling data in. GET parameters go directly into the URL, which means they're visible in browser history and server logs. This is fine for non-sensitive things like city names or date ranges.

You'll use POST when you need to send data to the API — for example, submitting a long text to an NLP model for analysis, or sending a large parameter set that won't fit cleanly in a URL. POST puts data in the request body, which keeps it out of the URL and allows much larger payloads.

08 Status codes you must know

Every HTTP response comes with a three-digit status code. This is the API's way of telling you: "Here's whether it worked — and if not, why." Knowing these will save you enormous amounts of debugging time.

CodeCategoryMeaningWhat you should do
200 Success Everything worked perfectly Parse the JSON and use the data
201 Created Your POST created something new Confirm and proceed
400 Bad Request Your request has a mistake Check your URL, params, and body
401 Unauthorized API key is missing or wrong Check your authentication header
403 Forbidden Authenticated but not allowed Check your subscription tier
404 Not Found That endpoint doesn't exist Double-check the URL path
429 Rate Limited Too many requests, slow down Add delays; check rate limit headers
500 Server Error Something broke on their end Wait and retry — not your fault
503 Service Unavailable API is down or overwhelmed Wait; check the service status page
The rule of thumb: 2xx means success. 4xx means something is wrong on your side. 5xx means something is wrong on their side. Always check the status code before you try to parse the response body.

Step 1 — Get your free API key

Go to openweathermap.org, sign up for a free account, and navigate to "API keys" in your dashboard. Copy the key — you'll paste it into your .env file shortly. The free tier allows 60 calls per minute, which is more than enough for everything in this project.

🚨 Let's pause here — never paste your API key directly into code

An API key is like a password. If you paste it directly into your Python file and push that file to GitHub, your key becomes public — and anyone can use it under your account, racking up charges or hitting your rate limits.

The safe approach: store your key in a file called .env and load it using the python-dotenv library. Your actual script then never contains the real key — just a reference to a variable name. We'll set this up properly below.

🟡 Write in file: weather_pipeline.py — Part B
# ─── Function: fetch weather for one city ─────────────────────────────────────

def fetch_weather(city_name):
    """
    Sends a GET request to the API for the given city.
    Returns a clean dictionary of weather data,
    or None if the request failed for any reason.
    """

    # Build the query parameters dictionary
    # The requests library will convert this into URL query params automatically
    params = {
        "q": city_name,
        # tells the API which city we want

        "appid": API_KEY,
        # our authentication key — sent with every request

        "units": "metric"
        # "metric" gives us Celsius; "imperial" would give Fahrenheit
    }

    # ─── Send the HTTP GET request ────────────────────────────────────────────────
    try:

        response = requests.get(
            BASE_URL,
            params=params,
            timeout=10
        )
        # requests.get() sends the actual HTTP request
        # BASE_URL is the address, params become the query string
        # timeout=10 means: give up if the server takes longer than 10 seconds

        # ─── Check the status code BEFORE trying to parse data ────────────────────

        if response.status_code == 200:
            # 200 means the request worked perfectly

            data = response.json()
            # .json() converts the response body from a JSON string into a Python dict
            # we can now access values using data["key"]["nested_key"]

            # Pull out only the fields we care about
            # This is called "flattening" the nested JSON into a flat record
            weather_record = {
                "city":           data["name"],
                # the city name as the API recognises it

                "country":        data["sys"]["country"],
                # country code, e.g. "KE" for Kenya

                "temperature_c":  data["main"]["temp"],
                # current temperature in Celsius

                "feels_like_c":   data["main"]["feels_like"],
                # perceived temperature accounting for humidity and wind

                "humidity_pct":   data["main"]["humidity"],
                # relative humidity as a percentage

                "description":    data["weather"][0]["description"],
                # "weather" is a list — [0] means the first (and usually only) item

                "wind_speed_ms":  data["wind"]["speed"]
                # wind speed in metres per second
            }

            return weather_record
            # return the clean record — not the messy raw response

        elif response.status_code == 401:
            # our API key is wrong, missing, or not yet activated
            print(f"[AUTH ERROR] Check your API key. Got 401 for {city_name}")
            return None

        elif response.status_code == 404:
            # the city name wasn't found in the API's database
            print(f"[NOT FOUND] City not recognised: {city_name}")
            return None

        elif response.status_code == 429:
            # we're sending requests too fast and got rate-limited
            print(f"[RATE LIMIT] Too many requests. Waiting 60 seconds...")
            time.sleep(60)
            return None

        else:
            # catch any other unexpected status code
            print(f"[ERROR] Status {response.status_code} for {city_name}")
            return None

    except requests.exceptions.Timeout:
        # the server took too long to respond
        print(f"[TIMEOUT] Request timed out for {city_name}")
        return None

    except requests.exceptions.ConnectionError:
        # no internet connection, or the URL is completely unreachable
        print(f"[CONNECTION ERROR] Check your internet connection.")
        return None

Part C — The pipeline loop and output

🟡 Write in file: weather_pipeline.py — Part C
# ─── Function: run the full pipeline ──────────────────────────────────────────

def run_pipeline():
    """
    Loops through each city in our CITIES list,
    fetches weather data, collects results,
    and saves everything to a CSV file.
    """

    all_results = []
    # we collect every successful weather record here before saving

    print("Starting weather pipeline...\n")

    for city in CITIES:

        print(f"Fetching: {city}")

        result = fetch_weather(city)
        # call our fetch function for this city
        # it returns either a dict (success) or None (failure)

        if result is not None:
            all_results.append(result)
            # only add to our list if the fetch succeeded

            print(
                f"  Success: {result['city']} — "
                f"{result['temperature_c']}C, {result['description']}"
            )

        else:
            print(f"  Skipped {city} — see error above")

        time.sleep(1)
        # wait 1 second between each city request
        # this is polite — it avoids hammering the free API and triggering rate limits

    # ─── Save results to CSV if we got anything ───────────────────────────────────

    if all_results:

        df = pd.DataFrame(all_results)
        # pd.DataFrame turns our list of dicts into a table
        # each dict becomes a row; each key becomes a column header

        df["fetched_at"] = pd.Timestamp.now()
        # add a timestamp column so we always know when this data was collected

        output_file = "weather_data.csv"

        df.to_csv(output_file, index=False)
        # save to CSV in the same folder as this script
        # index=False means don't write row numbers into the file

        print(f"\nDone. {len(df)} cities saved to {output_file}")
        print("\nQuick preview:")
        print(
            df[["city", "temperature_c", "humidity_pct", "description"]]
        )

    else:
        print("No data was collected. Check the errors printed above.")


# ─── Entry point ──────────────────────────────────────────────────────────────

if __name__ == "__main__":
    run_pipeline()
# this block only runs when you execute the file directly
# if this file were imported by another script, run_pipeline() would NOT auto-run
# this is a Python best practice for all scripts you write

💡 Here's what's going on under the hood with response.json()

When you call response.json(), the JSON string in the response body gets converted into a Python dictionary. That's why we access values with square brackets: data["main"]["temp"] goes into the outer dict, finds the "main" key, and then grabs the "temp" value inside that nested dictionary.

Notice that data["weather"] returns a list rather than a dict. That's why we write data["weather"][0]["description"] — the [0] means "give me the first item from the list," and then we grab the "description" key from that item.

When you get a KeyError, it means you're trying to access a key that doesn't exist in the response. The fix is always to print(data) first and look at the actual structure before accessing nested fields.
