01 What is an API? ✓Mark done

Before we touch a single line of code, let's build a real picture of what an API actually is. The letters stand for Application Programming Interface — and yes, that sounds like a lot. But the idea is simple enough that we'll get it in the next two minutes.

🔁 Think of it like a restaurant

Imagine you walk into a restaurant. You don't go into the kitchen and cook your own food. You sit down, look at a menu, and give your order to the waiter. The waiter goes to the kitchen, collects your food, and brings it back to you.

The waiter is the API. You are the application (your Python script). The kitchen is the server holding the data. The menu is the list of things you're allowed to ask for. You never need to know how the kitchen works — you just need to know how to talk to the waiter.

In technical terms: an API is a set of rules that lets one program talk to another. When your Python script sends a request to an API, the API's server receives it, processes it, and sends back a response — usually as data in a format called JSON.

💡 What is JSON, really?

JSON stands for JavaScript Object Notation. It's just a way of packaging up data so it can travel across the internet cleanly. It looks exactly like a Python dictionary. If you've written {"name": "Mel", "city": "Nairobi"} in Python — you already understand JSON. Curly braces, keys, and values. That's it.

02 Why data scientists need to know this ✓Mark done

Most real-world data doesn't sit in a nice CSV file waiting for you. It lives inside systems — and the only way to get it out is through an API. Here are situations you will actually run into:

  • Pulling weather data for a forecasting model
  • Fetching financial market prices for analysis
  • Pulling social media data for sentiment analysis
  • Sending data to an ML model hosted on a cloud service
  • Connecting your dashboard to a live data source
  • Automating data ingestion as part of a pipeline
Bottom line: APIs are how the world shares data. If you want to work with real, live data — you need to know how to request it, receive it, and handle it.

03 HTTP — the language of the web ✓Mark done

Before we can talk about REST APIs, we need to understand HTTP. This is the part most tutorials skip — and it's exactly why beginners feel confused later. Let's do it properly.

What is HTTP?

HTTP stands for HyperText Transfer Protocol. A "protocol" is just a set of rules for how two things communicate. HTTP is the rulebook that governs how computers talk to each other over the internet.

HTTP vs HTTPS — what the S means

You've seen both http:// and https:// in browser URLs. The S stands for Secure. HTTPS adds a layer of encryption — called TLS — that scrambles everything travelling between your computer and the server so no one in the middle can read it.

💡 Why APIs always use HTTPS — and why your API key depends on it

Every API call you make sends your API key in the request — either in the URL or in a header. If the connection is plain HTTP, that key travels as readable text across the internet. Anyone sitting on the same network (a café WiFi, a shared office connection) could intercept and steal it.

HTTPS encrypts everything in transit — your API key, your parameters, the response data. This is why all serious APIs require HTTPS and why you should never use a plain HTTP API for anything involving authentication or sensitive data.

Why HTTP theory matters

When something goes wrong — and it will — you need to read the error. HTTP errors have specific codes. If you don't know that 401 means "you didn't authenticate properly" or 429 means "too many requests," you'll waste hours debugging blind.

One more thing worth knowing now: HTTP is stateless — each request is completely independent. The server remembers absolutely nothing between calls. Every request must include your API key and all parameters every single time.

The request-response cycle

Every time you type a URL into your browser and hit Enter, this is exactly what happens behind the scenes. Understanding this cycle makes every API interaction feel familiar rather than mysterious.

🔁 What is DNS — the internet's phonebook

Your computer doesn't know where api.openweathermap.org lives. The internet has a phonebook called DNS (Domain Name System) that converts human-readable domain names into actual IP addresses — the real numerical location of the server. DNS runs silently in the background before your request even leaves your machine.

PartWhat it isReal example

What does a request body actually look like?

The table above says "JSON data" for the body — but what does that mean in practice? When you make a POST request (for example, sending text to an NLP model), the data you're sending travels in the body as a JSON string. Here's what that looks like:

POST request body — what travels inside it
# This is the Python dictionary you want to send
payload = {
    "text":     "The weather in Nairobi is perfect today.",
    "language": "en",
    "model":    "sentiment-v2"
}

# requests converts it to JSON and places it in the body automatically
response = requests.post(
    "https://api.nlpservice.com/v1/analyse",
    json=payload,
    # json= tells requests: convert this dict to JSON and put it in the body
    headers={"Authorization": f"Bearer {API_KEY}"}
)

See it before you code it

Before writing a single line of Python, paste this URL directly into your browser and hit Enter. You'll see the raw JSON response your script will eventually receive. This makes HTTP tangible — you can see exactly what the API sends back before you try to handle it in code.

Paste this into your browser address bar
https://api.openweathermap.org/data/2.5/weather?q=Nairobi&appid=YOUR_KEY&units=metric

Replace YOUR_KEY with your actual API key from openweathermap.org
You'll see raw JSON — the same data your Python script will receive

What is inside an HTTP response?

PartWhat it tells you
Status codeDid it work? 200 = yes, 404 = not found, 500 = server error
HeadersMetadata about the response (content type, rate limit info, etc.)

🚨 Hold on — what are "headers" exactly?

Headers are extra information attached to a request or response — like a sticky note on a letter. The most important header you'll use is the Authorization header — that's where your API key goes. Another common one is Content-Type: application/json, which tells the server the data you're sending is in JSON format. You don't set most headers manually — the requests library handles them — but you'll always add your API key header by hand.

04 REST — a set of rules for building APIs ✓Mark done

REST stands for Representational State Transfer. You don't need to memorise that. Just know REST is a style of building APIs — a set of rules that make an API predictable and easy to use.

🔁 Real world version

Imagine every post office in the world agreed to follow the same rules: packages always go in the same format, labels are always in the same position, tracking numbers always look the same. Once you've shipped a package once, you know how to ship from any post office anywhere.

REST is that agreement for APIs. Once you know how REST works, you can pick up almost any REST API in the world because they all follow the same patterns.

05 REST architecture — the six constraints ✓Mark done

1. Client-Server

The client (your script) and the server (the API) are completely separate. You can change your script without breaking the server, and vice versa.

2. Stateless

The server does not remember you between requests. Every single request must include all the information the server needs — your API key, your query, everything. This is why you include your authentication headers in every single call, not just the first one.

🚨 Stateless is actually a feature

Because the server doesn't have to remember anything, it can handle millions of simultaneous requests without getting confused. The practical impact: always include your auth header and all parameters in every request. Never assume the API "remembers" your last call.

3. Cacheable

Responses can be stored and reused instead of making the same request again. In data pipelines, this matters when you're calling the same endpoint repeatedly — caching can make your work faster and cheaper if the API charges per call.

4. Uniform Interface

Every REST API uses the same structure: standard HTTP methods, standard URL patterns, and standard response formats. Once you know one REST API, you can pick up another very quickly.

💡 What "uniform interface" means in practice

Two completely different APIs — weather data and financial data — both follow REST. Both use GET to fetch, POST to send, JSON in responses, and standard status codes. You don't learn an entirely new system for each API. The interface is uniform across all of them.

5. Layered System

Your request might pass through load balancers, caches, and security gateways. REST allows this — your code stays exactly the same regardless of what's in the middle.

6. Code on Demand (optional)

Servers can optionally send executable code back to the client. Rarely relevant in data science work.

The two you'll think about most: Stateless and Uniform Interface. These shape every API call in a data pipeline.

06 Anatomy of an API request ✓Mark done

API URL — fully annotated
https://api.openweathermap.org/data/2.5/weather?q=Nairobi&appid=YOUR_KEY&units=metric

https://                       the protocol — always HTTPS for secure APIs
api.openweathermap.org         the host / domain of the API server
/data/2.5/weather              the endpoint path — the specific resource we want
?                              marks the start of query parameters
q=Nairobi                      query param: city we want weather for
&appid=YOUR_KEY                query param: our API key for authentication
&units=metric                  query param: return temperature in Celsius

Base URL vs endpoint path

The base URL is the root address: https://api.openweathermap.org. The endpoint path is what you add: /data/2.5/weather. In your code you'll store the base URL as a constant and build the full URL dynamically.

07 HTTP methods — the verbs of the web ✓Mark done

MethodWhat it doesEveryday analogyHas body?
GETFetch / read dataReading a menuNo
POSTSend data / create somethingPlacing an orderYes

PATCHUpdate part of somethingChanging just the drinkYes

💡 GET vs POST for data scientists

You'll use GET the most — you're usually pulling data in. GET parameters go directly into the URL, which is fine for non-sensitive things like city names.

Use POST when you need to send data to the API — submitting text to an NLP model, or sending a large parameter set that won't fit in a URL. POST puts data in the request body, which keeps it out of the URL and allows much larger payloads.

09 Project: Weather Data Pipeline ✓Mark done

Hands-on Project

Build a Real Weather Data Fetcher

We'll use the OpenWeatherMap API to pull live weather data for multiple African cities and save it to a CSV — exactly how a data pipeline step works in production.

Step 1 — Get your free API key

Go to openweathermap.org, sign up, and navigate to "API keys" in your dashboard. Copy the key. The free tier allows 60 calls per minute. New keys can take up to 10 minutes to activate.

Step 2 — Set up a virtual environment

A virtual environment is an isolated box for your project's libraries so they don't clash with other projects. Always create one before installing anything.

Step 3 — Install the libraries

🟢 Install dependencies
pip install requests python-dotenv pandas

Step 4 — Create your .env and .gitignore files

Your API key is a secret. Store it in a .env file and tell Git to never track it using .gitignore — so it never ends up on GitHub even by accident.

🟡 Create file: .env
# Replace the value below with your actual API key — no quotes needed
OPENWEATHER_API_KEY=paste_your_actual_key_here
🟡 Create file: .gitignore
# These files will never be tracked by Git
.env
__pycache__/
venv/
Why .gitignore matters: without it, pushing to GitHub makes your API key public. These three lines take 30 seconds and protect you permanently.

Step 5 — Part A: Imports and setup

🟡 weather_pipeline.py — Part A
import requests
import pandas as pd
import time
import os
from dotenv import load_dotenv

load_dotenv()
# reads the .env file and makes its values available

API_KEY = os.getenv("OPENWEATHER_API_KEY")

if not API_KEY:
    raise ValueError("API key not found. Check your .env file.")

BASE_URL = "https://api.openweathermap.org/data/2.5/weather"

CITIES = ["Nairobi", "Lagos", "Accra", "Kampala", "Dar es Salaam"]

Step 6 — Part B: The fetch function

🟡 weather_pipeline.py — Part B
def fetch_weather(city_name):
    params = {"q": city_name, "appid": API_KEY, "units": "metric"}

    try:
        response = requests.get(BASE_URL, params=params, timeout=10)

        if response.status_code == 200:
            data = response.json()
            return {
                "city":          data["name"],
                "country":       data["sys"]["country"],
                "temperature_c": data["main"]["temp"],
                "feels_like_c":  data["main"]["feels_like"],
                "humidity_pct":  data["main"]["humidity"],
                "description":   data["weather"][0]["description"],
                "wind_speed_ms": data["wind"]["speed"]
            }
        elif response.status_code == 401:
            print(f"[AUTH ERROR] Check your API key. Got 401 for {city_name}")
        elif response.status_code == 404:
            print(f"[NOT FOUND] City not recognised: {city_name}")
        elif response.status_code == 429:
            print(f"[RATE LIMIT] Waiting 60 seconds...")
            time.sleep(60)
        else:
            print(f"[ERROR] Status {response.status_code} for {city_name}")
        return None

    except requests.exceptions.Timeout:
        print(f"[TIMEOUT] Request timed out for {city_name}")
        return None
    except requests.exceptions.ConnectionError:
        print(f"[CONNECTION ERROR] Check your internet connection.")
        return None

Step 7 — Part C: The pipeline loop

🟡 weather_pipeline.py — Part C
def run_pipeline():
    all_results = []
    print("Starting weather pipeline...\n")

    for city in CITIES:
        print(f"Fetching: {city}")
        result = fetch_weather(city)

        if result is not None:
            all_results.append(result)
            print(f"  Done: {result['city']} — {result['temperature_c']}C, {result['description']}")
        else:
            print(f"  Skipped {city}")

        time.sleep(1)

    if all_results:
        df = pd.DataFrame(all_results)
        df["fetched_at"] = pd.Timestamp.now()
        df.to_csv("weather_data.csv", index=False)
        print(f"\nDone. {len(df)} cities saved to weather_data.csv")
        print(df[["city", "temperature_c", "humidity_pct", "description"]])
    else:
        print("No data collected. Check errors above.")


if __name__ == "__main__":
    run_pipeline()

Step 8 — Run it

🟢 Run in Terminal
python weather_pipeline.py
This is a production-quality data pipeline. It authenticates, fetches, handles errors, and saves results. This exact pattern is what data engineers write every day.
